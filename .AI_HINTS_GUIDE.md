# AI Hints Guide for UniFi Camera Proxy

This guide provides context and patterns to help AI assistants work effectively with this codebase.

## Project Overview

**Purpose**: Enables non-Ubiquiti cameras (especially Frigate) to work within the UniFi Protect ecosystem via WebSocket protocol emulation.

**Key Components**:
- **`unifi/main.py`**: Entry point, CLI argument parsing, camera factory
- **`unifi/cams/base.py`**: Core camera base class with protocol handling, snapshot management, event lifecycle
- **`unifi/cams/frigate.py`**: Frigate-specific integration via MQTT
- **`unifi/cams/managers/`**: Event lifecycle managers (SmartDetectEventManager, SmartMotionEventManager)
- **`unifi/cams/handlers/`**: Protocol handlers for snapshots, video streams, etc.
- **`unifi/cams/frigate_lib/`**: Frigate-specific utilities (descriptors, zones, snapshots, motion)

## Architecture Patterns

### 1. Mixin Composition Pattern

```python
class UnifiCamBase(ProtocolHandlers, VideoStreamHandlers, SnapshotHandlers, metaclass=ABCMeta):
    """Composition root for UniFi Protect camera emulation."""
```

**Pattern**: Multiple mixin classes provide orthogonal functionality. The base class doesn't call `super().__init__()` through MRO, so state is set directly.

**AI Hint**: When modifying behavior, add methods to the appropriate mixin rather than changing the base class.

### 2. Event Lifecycle Delegation

```python
# Base class owns lifecycle managers
self._smart_events = SmartDetectEventManager(...)
self._smart_motion = SmartMotionEventManager(...)

# Subclasses call trigger_* methods
await self.trigger_smart_detect_start(object_type=SmartDetectObjectType.PERSON)
```

**Pattern**: Event lifecycle is delegated to managers; base class provides thin delegation layer for backward compatibility.

**AI Hint**: Don't change the public `trigger_*` API signatures in base class—add new functionality internally.

### 3. Frigate Integration Layers

```
FrigateCam (MQTT subscriber)
  ↓
FrigateEventHandlerMixin (protocol messages)
  ↓
RTSPCam (video streaming)
  ↓
UnifiCamBase (core protocol)
```

**Pattern**: Multiple inheritance with clear separation of concerns:
- **MQTT layer**: Receives Frigate events
- **Protocol layer**: Translates to UniFi messages
- **Streaming layer**: Handles RTSP/video

**AI Hint**: When adding Frigate features, add to `FrigateCam` or `FrigateEventHandlerMixin`, not base classes.

## Key Data Structures

### EventSmartDetect Message Structure

```python
{
    "from": "ubnt_avclient",
    "functionName": "EventSmartDetect",
    "inResponseTo": 0,
    "messageId": <monotonic counter>,
    "payload": {
        "clockMonotonic": <ms since boot>,
        "clockStream": <stream-relative ms>,
        "clockStreamRate": 1000,
        "clockWall": <epoch ms>,
        "eventId": <shared counter>,
        "edgeType": "enter|moving|leave|none",
        "objectTypes": ["person"],
        "displayTimeoutMSec": 300,
        "descriptors": [...],
        "zonesStatus": {"1": {"level": 75, "status": "enter"}},
        "smartDetectSnapshotFullFoV": "<path>",
        "smartDetectSnapshots": [...],
        "trackerIDAttrMap": {"700000": {"objectType": "person", "zone": [1]}}
    },
    "responseExpected": false,
    "timeStamp": "<ISO timestamp>",
    "to": "UniFiVideo"
}
```

**AI Hint**: `messageId` is shared across ALL message types per connection. `eventId` is per-camera but shared across all trackers.

### Descriptor Object

```python
{
    "trackerID": 700000,
    "objectType": "person",
    "coord": [x, y, w, h],  # in 640x360 sub-stream space
    "boxColor": "red|white",
    "confidenceLevel": 85,
    "firstShownTimeMs": <epoch>,
    "idleSinceTimeMs": 0,  # = firstShownTimeMs if stationary
    "stationary": false,
    "zones": [1, 2],  # most-recent-first
    "attributes": null,
    "lines": null,
    "loiterZones": null,
    "name": "",  # populated if face recognition matched
    "tag": ""
}
```

**AI Hint**: `coord` uses low-res sub-stream coordinates, not full-frame. Scale accordingly.

## Protocol-Specific Notes

### Clock Timing

- `clockMonotonic`: Elapsed ms since emulator start (consistent across all messages)
- `clockStream`: Fixed offset from `clockMonotonic` (~20-21s behind)
- `clockWall`: Epoch ms from Frigate's `frame_time * 1000`

**AI Hint**: Maintain a single `clockMonotonic` counter; don't reset per event.

### Zone Status Completeness

Every message must include ALL configured zones, not just active ones:

```python
zonesStatus = {}
for zone_id in configured_zone_ids:
    zonesStatus[str(zone_id)] = {
        "level": zone_level,  # 0-100
        "status": "enter|moving|leave|none"
    }
```

**AI Hint**: Don't omit zones—Protect expects all configured zones on every message.

### Stationary Object Handling

```python
if after.position_changes == 0:
    # Stationary object
    edgeType = "none"
    objectTypes = []  # Clear for stationary
    boxColor = "white"
```

**AI Hint**: Stationary objects have `edgeType: "none"` and don't populate `objectTypes`.

## AI-Specific Tips

1. **Don't modify base class `__init__`**: Use dependency injection or add optional parameters
2. **Preserve API signatures**: The `trigger_*` methods are stable; add complexity internally
3. **Use existing enums**: `SmartDetectObjectType` is the source of truth for object types
4. **Zone IDs are static**: Once configured in Protect, they don't change—use the `zone_map` config
5. **Message IDs are global**: One counter for all message types per connection
6. **Event IDs are per-camera**: Increment independently for each camera session
7. **Snapshots are cached**: Prefer cached event-specific snapshots over live camera snapshots
8. **Clock consistency**: Maintain monotonic clocks across all messages
9. **All zones on all messages**: Never omit zones from `zonesStatus`
10. **Stationary vs. active**: Distinguish via `edgeType` and `objectTypes` fields
11. **Rate-limit cleanup operations**: `_cleanup_old_events()` runs on every `trigger_*` call—don't call it repeatedly
12. **Cancel tasks before awaiting**: In `smart_motion_manager.py`, cancel pulse task BEFORE start task to prevent race conditions
13. **Task cancellation pattern**: Always `cancel()` → `await` → `except CancelledError` for background tasks
14. **Use `asyncio.TaskGroup` for related tasks**: FrigateCam uses this for MQTT handlers—wrap related tasks together
15. **Event ID collision prevention**: Use `int(time.time() * 1000) + (counter % 1000)` to avoid collisions across seconds
16. **Snapshot path race conditions**: Only one path should set the final snapshot path—use atomic operations
17. **Box color derivation**: `"red"` when `edgeType != "none"`, `"white"` for stationary/background
18. **Stationary bystanders in leave messages**: Include in `descriptors[]` only, NOT in `smartDetectSnapshots[]` or `trackerIDAttrMap`
19. **Pulse event eventId**: Always use `18446744073709551615` (max uint64) for motion pulse events
20. **Clock timing**: `clockBestMonotonic/clockBestWall` = `0` for start/pulse, actual start time for stop

## Concurrency Patterns

### Background Tasks

All background tasks (pulse loops, timeout monitors) must be:
1. Created with `asyncio.create_task()`
2. Stored in instance variables for tracking
3. Cancelled with `task.cancel()` before awaiting
4. Awaited to ensure cleanup

Example:
```python
self._pulse_task = asyncio.create_task(self._pulse_loop(event_id))

# Later, when stopping:
if self._pulse_task and not self._pulse_task.done():
    self._pulse_task.cancel()
    try:
        await self._pulse_task
    except asyncio.CancelledError:
        pass
```

### Task Groups

For multiple concurrent tasks (FrigateCam):
```python
async with asyncio.TaskGroup() as tg:
    tg.create_task(self.monitor_event_timeouts())
    tg.create_task(self.detect_fps_tracker.refresh_periodically())
    async for message in client.messages:
        tg.create_task(self.handle_detection_event(message))
```

**Warning**: TaskGroup will raise if any task fails. Catch exceptions and decide whether to continue.

### Task Cancellation Ordering

**Critical**: In `smart_motion_manager.py`, the pulse task must be cancelled BEFORE the start task:

```python
# WRONG - can cause race condition
if self._analytics_start_task and not self._analytics_start_task.done():
    self._analytics_start_task.cancel()
    await self._analytics_start_task

if self._motion_pulse_task and not self._motion_pulse_task.done():
    self._motion_pulse_task.cancel()
    await self._motion_pulse_task

# CORRECT
if self._motion_pulse_task and not self._motion_pulse_task.done():
    self._motion_pulse_task.cancel()
    await self._motion_pulse_task

if self._analytics_start_task and not self._analytics_start_task.done():
    self._analytics_start_task.cancel()
    await self._analytics_start_task
```

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│   Motion Start  │────▶│   Motion Pulse  │────▶│   Motion Stop   │
│  (linger delay) │     │     (heartbeat) │     │     (cleanup)   │
└─────────────────┘     └─────────────────┘     └─────────────────┘
       │                        │                        │
       │                        │                        │
       ▼                        ▼                        ▼
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│ Smart Detect    │◀────│ Smart Detect    │────▶│ Smart Detect    │
│   (enter/move)  │     │    (enter/move) │     │    (leave)      │
└─────────────────┘     └─────────────────┘     └─────────────────┘
```

**Key Points**:
- Motion window can contain multiple Smart Detect events
- Smart Detect events are independent but linked to motion window
- Cleanup only happens on motion stop

## Common Pitfalls

### Pitfall 1: Event ID Collisions

**Problem**: Using `time.time() * 1000` for event IDs can cause collisions if events span seconds.

**Solution**: Use a monotonic counter:
```python
self._event_id_counter = 0
event_id = int(time.time() * 1000) + (self._event_id_counter % 1000)
self._event_id_counter += 1
```

### Pitfall 2: Snapshot Path Race Conditions

**Problem**: Snapshot paths are set in multiple places (smart detect stop, motion stop).

**Solution**: Use atomic operations or ensure only one path sets the final value.

### Pitfall 3: Zone Status Completeness

**Problem**: Forgetting to include all zones in `zonesStatus`.

**Solution**: Always iterate over ALL configured zones, not just active ones:
```python
zonesStatus = {}
for zone_id in configured_zone_ids:
    zonesStatus[str(zone_id)] = {
        "level": zone_level,
        "status": zone_status  # "enter", "moving", "leave", "none"
    }
```

### Pitfall 4: Clock Desynchronization

**Problem**: `clockMonotonic` and `clockWall` must stay in sync.

**Solution**: Update both on every message, never reset `clockMonotonic`.

### Pitfall 5: Cleanup Loop

**Problem**: `_cleanup_old_events()` runs on every `trigger_*` call, which can cause events to be deleted mid-lifecycle.

**Solution**: Add a cooldown to prevent cleanup on every call:
```python
self._cleanup_cooldown_until = 0.0

def _cleanup_old_events(self) -> None:
    current_time = time.time()
    if current_time < self._cleanup_cooldown_until:
        return  # Rate-limited
    
    self._cleanup_cooldown_until = current_time + 1.0  # 1 second cooldown
    # ... rest of cleanup
```

### Pitfall 6: Task Cancellation Race

**Problem**: Cancelling start task before pulse task can cause the start task to send an event that should have been cancelled.

**Solution**: Always cancel pulse task first, then start task.
